# 第 1 周你要掌握什么

这周只学 4 件事：

- **消息结构**：system、user、assistant 到底各管什么
- **结构化输出**：为什么不能让模型自由发挥
- **function calling / tool calling 基本认知**：模型输出不仅可以是文本，还可以是“调用某种能力的意图”
- **后端封装**：把上面这些变成一个稳定 API

Anthropic 的当前提示工程主文档把清晰指令、示例、XML/结构化组织、tool use、agentic systems 放在同一主线上；OpenAI 的当前 Responses API 文档把 text I/O、Structured Outputs、Function calling、conversation state、tools 放在一个接口体系里；Hugging Face 的 LLM Course 第 1 章适合作为你补“LLM 基本概念”的轻量入口。

## 周三，1 小时

## 目标：把理论翻成你自己的业务语言

### 做什么

把你已有“提槽”经验翻译成标准 AI 开发语言。

在 `docs/slot-extraction-design.md` 里写这 4 段：

### 1. 任务定义

例如：

- 输入：用户自然语言 query
- 输出：意图 + 槽位 + 置信度 + 原始 query

### 2. 为什么需要结构化

例如：

- 后续要路由工具
- 后续要做搜索条件拼装
- 后续要做过滤和生成控制

### 3. 哪些字段必须稳定

例如：

- intent
- slots
- confidence
- missing_required_slots

### 4. 错误输出怎么处理

例如：

- 无法识别意图
- 多意图冲突
- 槽位不完整
- query 噪音过大

### 今天的验收标准

你写出来的这份文档，不能出现“感觉”“差不多”“大概是”这种模糊词。

## 周四，2 小时

## 目标：把提槽能力变成后端接口

### 第一段，40 分钟

定义 Pydantic schema。

建议至少有这两个：

```
class SlotExtractionRequest(BaseModel):
    query: str

class SlotExtractionResponse(BaseModel):
    intent: str
    slots: dict
    confidence: float
    missing_required_slots: list[str]
    raw_query: str
```

### 第二段，50 分钟

要求：

- 入参固定

- 即使模型输出乱了，也要有兜底逻辑

### 今天的关键原则

`/extract` 是主角，`/chat` 只是陪跑。

### 今天的验收标准

- `/extract` 能返回标准 JSON

## 周五，1 小时

## 目标：第一次做 Prompt 模板化

### 做什么

在 `app/prompts/` 下新建两个文件：

- `slot_extraction_system.txt`
- `slot_extraction_user.txt`

### system prompt 里要有的内容

- 你的角色
- 任务目标
- 输出约束
- 禁止行为
- 失败时的默认策略

### user prompt 里要有的内容

- 原始 query
- 输出 schema 提示
- 必要时加 1–2 个 few-shot 示例

### 你今天要做的测试

用 5 条 query 分别比较：

- 没有 schema 约束
- 有 schema 约束

看输出稳定性是不是更高。

Anthropic 的提示工程文档明确把清晰指令、示例、结构化组织和 consistency 放在同一套方法里；它也单独有“提高一致性”的指导。

### 今天的验收标准

你能明确说出：

- schema 约束具体提升了什么
- few-shot 是否真的必要
- 哪种 prompt 写法最稳定

# 第一周结束后的判断标准

如果第一周学对了，你会出现这几个变化：

### 以前

- 写 prompt 靠感觉
- 返回结果 mostly 是自然语言
- 后端拿模型输出像拆盲盒

### 现在

- 知道 prompt 要像协议设计
- 知道结构化输出是第一优先级
- 知道后端接口该如何包住模型的不确定性

这一步看着朴素，但很关键。
 很多 Agent 项目后面越做越像沼泽，不是因为框架不行，而是第一层接口契约压根没立住，后面全在泥地上盖楼。
