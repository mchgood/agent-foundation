# Anthropic Prompting Best Practices 学习笔记

## 资料来源

- Anthropic 官方文档：<https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/system-prompts>
- 页面标题：`Prompting best practices`

## 学习目标

- 理解 Prompt 工程为什么不是“会提问”而是“定义模型工作方式”
- 掌握 system prompt、上下文、示例、结构化标签的作用分工
- 了解长上下文、工具调用、thinking、agentic systems 在同一条工程主线里的位置
- 能把文档中的原则转成你自己的 Prompt 模板

## 笔记概要

这篇文档是 Anthropic 对 Claude 提示工程的总说明书。它不是只讲“怎么写提示词”，而是从基础提示原则一路讲到输出控制、工具使用、思考配置和 Agent 系统设计。  
对初学者来说，最重要的不是记住所有技巧，而是建立一个总判断：

**Prompt 的作用是给模型设定任务边界、输入结构、行为模式和输出合同。**

## 详细笔记

### 1. 文档整体结构

这篇文档大致分为五块：

1. `General principles`
2. `Output and formatting`
3. `Tool use`
4. `Thinking and reasoning`
5. `Agentic systems`

这说明 Anthropic 并不把 Prompt 工程看成单独技能，而是把它放在完整应用链路中理解。  
也就是说，Prompt 不只是决定“回答写成什么样”，还决定：

- 模型是否会使用工具
- 模型会不会过度推理
- 多步任务怎么拆解
- Agent 在安全性和自主性之间如何平衡

### 2. General principles：最基础也最重要

#### 2.1 Be clear and direct

文档首先强调：Claude 对明确、直接、无歧义的指令响应更好。  
如果你需要某种输出格式、完整度或执行力度，不要让模型自己推断。

这里有一个非常重要的判断标准：  
如果一个完全不了解背景的同事看了你的 Prompt 都会困惑，那么模型大概率也会困惑。

这一节想教你的不是“多写字”，而是“消除歧义”。  
例如，“帮我分析一下”就很弱，因为没有说明：

- 分析什么
- 分析到什么粒度
- 输出给谁看
- 结果用什么格式表达

你以后写 Prompt 时，至少要补齐这四类信息。

#### 2.2 Add context to improve performance

文档强调，不只是“命令”重要，“为什么这样要求”也重要。  
这意味着上下文不是附属信息，而是帮助模型理解目标的约束条件。

比如：

- 不要只说“不要使用省略号”
- 要说“这段内容会被 TTS 朗读，所以不要使用省略号”

后者更有效，因为它告诉模型“这个规则背后的使用场景是什么”。  
这类解释会让模型更容易做正确泛化，而不是机械遵守某一句局部规则。

#### 2.3 Use examples effectively

Anthropic 明确把 examples 视为最稳定的控制手段之一。  
示例的价值不只是“举例”，而是在向模型展示：

- 理想输出长什么样
- 哪些模式应该保留
- 边界情况如何处理

文档建议示例应该满足三点：

- 和真实任务接近
- 有一定多样性
- 与指令和输入清晰分隔

还提到通常 3 到 5 个示例效果较好。  
这对于你以后做学习总结、槽位抽取、结构化提取非常重要，因为很多一致性问题靠“重复要求”解决不了，靠示例反而更稳。

#### 2.4 Structure prompts with XML tags

官方推荐使用 XML tags 组织复杂 Prompt，例如：

- `<instructions>`
- `<context>`
- `<examples>`
- `<input>`

这里的重点不是 XML 语法本身，而是信息分层。  
当 Prompt 同时包含任务规则、背景材料、少样本和用户输入时，如果没有明确边界，模型很容易混淆哪些是规则、哪些是样例、哪些是待处理输入。

因此，这一节真正想教你的能力是：

**把 Prompt 组织成结构化文档，而不是一大段自然语言。**

#### 2.5 Give Claude a role

文档建议在 system prompt 中给模型一个明确角色，比如“你是一个 Python 编程助手”。  
角色不是装饰，它会影响模型优先关注什么、如何措辞、怎样选择解释深度。

对于学习场景，你可以把角色写成：

- AI 学习教练
- 知识整理助手
- 面向初学者的解释型助手

角色越清楚，模型越容易维持一致行为。

### 3. Long context prompting：长上下文的核心不是塞更多材料

这一节非常值得反复看，因为后面做 RAG、多文档分析、Agent 都会用到。

文档提了几个关键策略：

#### 3.1 长材料放前面，问题放后面

官方指出，在 20k+ token 的复杂输入里，把长文档放前面、查询放在后面，效果通常更好。  
这和很多初学者的直觉相反。很多人会先提问题，再把资料附在后面。

Anthropic 这里其实是在告诉你：  
Prompt 不只是“说什么”，还包括“信息出现的顺序”。

#### 3.2 多文档要做显式边界

不要简单拼接多份资料，而要给每份资料单独包裹结构和元数据，例如：

- 文档编号
- 来源
- 内容区域

这能帮助模型保持“材料视图”的清晰度，减少不同来源内容被混淆。

#### 3.3 先抽引用，再做分析

文档建议在长文档任务中，先让模型提取相关原句，再基于这些原句做总结、诊断或建议。  
这是一种非常工程化的思路，因为它把“证据抽取”和“结论生成”拆成了两个阶段。

这样做能降低模型在长材料中胡乱泛化的概率。

### 4. Output and formatting：输出控制不是最后修辞，而是设计输入时就要考虑

这一部分讲的是如何更稳定地控制输出形态。

#### 4.1 用正向要求代替负向禁止

文档建议多说“你应该怎么写”，少说“你不要怎么写”。  
例如，不要说“不要用 markdown”，而是说“请用连贯自然段书写”。

原因很简单：  
负向限制只给了边界，没有给目标；正向要求同时给了边界和目标。

#### 4.2 Prompt 自身风格会影响输出风格

如果你的 Prompt 充满 markdown、标题、列表，模型也更可能用类似风格作答。  
如果你想得到纯 prose，就让 Prompt 本身也更接近 prose。

这对你以后写学习助手 Prompt 很重要，因为很多格式问题不是输出端单独的问题，而是输入端已经在示范某种风格。

#### 4.3 新模型更简洁，必要时显式要求过程总结

Anthropic 提到，新模型会更直接，不一定在每次工具调用后都写一段解释。  
如果你想让它汇报操作过程，就必须在 Prompt 中显式提出。

这提醒你：  
模型“没说”不一定是没做，而可能是默认风格更简洁。

#### 4.4 数学和技术表达可能默认偏向 LaTeX

文档还提到某些新模型在技术解释中默认倾向 LaTeX。  
如果你希望纯文本表达，需要主动约束。

这里体现的是一个核心原则：  
输出风格不只是内容问题，还包括符号系统和展示方式。

### 5. Tool use：模型何时行动，必须由你定义

这一部分非常实用。  
Anthropic 讨论的不是“工具能不能调用”，而是“什么时候该调用工具，什么时候只给建议”。

你可以把 system prompt 设计成两种模式：

- 默认主动执行
- 默认保守，只在明确要求时才执行

这直接影响 Agent 的行为风格。  
文档还提醒，新模型对 system prompt 更敏感，所以以前那种为了逼模型动手而写得很激进的提示词，现在可能会造成过度行动。

这说明随着模型变强，Prompt 不一定要更激进，反而要更细致地控制边界。

### 6. Thinking and reasoning：推理不是越多越好

Anthropic 指出，Claude Opus 4.6 在较高 `effort` 下会做更多前置探索。  
这通常能提升复杂任务质量，但也会带来：

- thinking token 增长
- 响应时间变长
- 可能出现过度搜索、过度工具调用

因此，文档建议：

- 不要默认鼓励“更全面、更彻底”
- 简单问题不要强行启用复杂 thinking
- 可以让模型在结束前做自检

这背后的原则很关键：  
推理是资源，不是越多越好，而是要与任务复杂度匹配。

### 7. Agentic systems：Prompt 最终要进入系统设计

文档后半部分进入 Agent 系统主题，例如：

- 长周期任务管理
- 状态追踪
- 多上下文窗口协作
- 自主性与安全性平衡
- 研究与信息收集
- 子代理协作
- 减少无意义文件创建
- 避免为了通过测试而硬编码
- 降低 hallucination

这一部分对你的启发应该是：

Prompt 不是应用的表面层，而是 Agent 行为协议的一部分。  
如果你以后要做真正的 AI 学习助手，system prompt、工具定义、状态管理、输出格式都必须一起设计。

## 关键术语

- `system prompt`：定义模型全局角色和行为边界的系统级指令
- `few-shot`：通过少量示例约束模型输出模式的方法
- `XML tags`：用结构化标签组织 Prompt 中不同信息区块的方式
- `long context`：大规模上下文输入场景
- `tool use`：模型决定何时调用外部工具的能力
- `thinking effort`：模型在回答前投入的推理深度或探索强度
- `agentic systems`：具备多步规划、工具调用和状态管理能力的智能系统

## 练习题

1. 为什么文档强调“清晰直接”比“优雅自然”更重要？请结合一个坏 Prompt 和一个改进版本说明。
2. 什么情况下应该补充上下文，而不是只写命令？
3. few-shot 示例为什么通常比反复强调要求更稳？
4. 为什么 XML tags 的重点不在 XML，而在“信息分层”？
5. 长文档任务中，为什么建议先抽取引用再做分析？
6. 输出控制部分为什么建议多写“该做什么”，少写“不要做什么”？
7. 在 Agent 场景里，system prompt 为什么必须明确“默认行动”还是“默认建议”？
8. `thinking` 为什么不能默认拉满？它会带来哪些代价？

## 学完后的自测标准

- 你能解释 system prompt、context、examples、input 的分工
- 你能自己写一个带 XML tags 的学习助手 Prompt
- 你能说清楚 Prompt 工程为什么会和 tool use、thinking、Agent 设计连在一起
