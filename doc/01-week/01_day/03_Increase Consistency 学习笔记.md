# Increase Consistency 学习笔记

## 资料来源

- Anthropic 官方文档：<https://docs.anthropic.com/en/docs/test-and-evaluate/strengthen-guardrails/increase-consistency>
- 页面标题：`Increase output consistency`

## 学习目标

- 理解“一致性”在 LLM 应用中具体指什么
- 掌握官方给出的几种稳定输出的方法
- 分清什么场景靠 Prompt，什么场景应该直接上 Structured Outputs
- 学会把示例、检索、角色和任务拆解结合起来提升复现度

## 笔记概要

这篇文档的核心任务是回答一个非常实际的问题：  
**为什么模型同样的任务有时输出很稳，有时又会漂移？**

Anthropic 给出的答案不是单一技巧，而是一套方法组合：先定义清楚输出结构，再用示例约束风格，用检索固定知识上下文，用任务拆链降低复杂度，用系统提示固定角色。如果目标是严格 JSON 结构，则应该直接使用 Structured Outputs。

## 详细笔记

### 1. 什么叫“一致性”

在 LLM 场景里，一致性不是抽象词，它通常指下面几类现象是否稳定：

- 输出格式是否稳定
- 输出风格是否稳定
- 字段结构是否稳定
- 多轮对话中的口径是否稳定
- 相同任务的处理步骤是否稳定

比如，你让模型做用户反馈总结：

- 第一次按 4 个字段输出
- 第二次变成一大段 prose
- 第三次又多了两个你没要的字段

这就是一致性问题。

### 2. 文档开头最重要的边界判断

Anthropic 在这篇文档里首先讲清了一件事：

如果你的目标是“输出始终严格符合某个 JSON Schema”，那么不要只靠 Prompt engineering，而应该直接用 `Structured Outputs`。

这说明一致性至少分两类：

- 行为一致性：回答方式、风格、步骤是否稳定
- 结构一致性：返回格式是否严格符合 schema

前者主要靠 Prompt、示例和流程设计；后者必须靠结构化约束能力。

这个判断非常重要，因为很多初学者会陷入一个误区：  
明明是严格结构问题，却不停加 Prompt 细节，结果还是不稳。

### 3. 方法一：明确定义输出格式

文档建议明确指定输出格式，可以是：

- JSON
- XML
- 自定义模板

重点不是“给个大概方向”，而是尽量减少解释空间。  
你写得越模糊，模型可自由发挥的空间越大，结果就越容易漂。

例如：

- 差的写法：请帮我总结这段用户反馈
- 更好的写法：请输出“问题概述、用户情绪、影响范围、建议动作”四部分

这里的关键思想是“压缩输出空间”。  
Prompt 的很多稳定性，本质上来自于你先把不必要的自由度关掉。

### 4. 方法二：Prefill 的历史位置与当前替代方案

文档明确说明，`prefill` 作为旧技巧已经被弃用，并且不支持 Claude Opus 4.6、Claude Sonnet 4.6、Claude Sonnet 4.5。

这部分你需要学到两个结论：

1. 不要继续把旧时代技巧当成当前主路径
2. 新模型下，应该优先选择：
   - Structured Outputs
   - 更清晰的 system prompt
   - 更明确的格式说明

这对你非常重要，因为网上很多旧教程还在教“预填 assistant 前缀”，但官方已经明确提示这不再是推荐方向。

### 5. 方法三：用示例提升输出复现度

文档指出，提供输出示例是提升一致性最有效的方法之一。  
这是因为示例同时约束了多个维度：

- 格式
- 语气
- 信息粒度
- 任务边界
- 特殊情况处理方式

如果你只靠自然语言要求，模型需要自己推断“合格答案长什么样”；  
如果你给了示例，模型就能直接对齐目标模式。

对于你当前的学习路径，这一点可以直接迁移到：

- 学习笔记整理
- 提槽接口
- 资料总结
- 问答结果格式化

建议你以后练习 few-shot 时，至少覆盖：

- 正常案例
- 边界案例
- 错误案例

### 6. 方法四：用检索固定知识上下文

Anthropic 特别提到，对 chatbot 或知识问答系统，如果你希望输出基于固定知识集，就应该使用 retrieval。

为什么 retrieval 会提升一致性？  
因为很多时候“不一致”不是风格问题，而是知识来源在漂。

如果模型有时根据你给的材料回答，有时又补进自己训练中的泛化知识，就容易出现：

- 这次引用 A
- 下次换成 B
- 再下一次口径混合

因此，retrieval 的作用不仅是补知识，更是在做“知识定锚”。

以后你如果基于 `doc/` 目录做学习助手，就应该让模型优先基于检索出的文档作答，而不是每次自由联想。

### 7. 方法五：把复杂任务拆成链式步骤

文档建议用 Prompt chaining 处理复杂任务。  
原因是复杂任务经常包含多个不同认知阶段，例如：

- 阅读
- 提取
- 分类
- 判断
- 生成
- 格式化

如果把这些全部塞进一步，模型就可能在任何中间阶段开始漂。  
而把任务拆开后，每一步都更专注，也更容易验证。

你可以把复杂任务拆成：

1. 先抽取信息
2. 再归类
3. 再生成结论
4. 最后做结构化输出

这对后面做 Agent 非常关键，因为多步任务的一致性往往来自于流程设计，而不是单条超级 Prompt。

### 8. 方法六：保持角色一致

文档最后讲角色一致性，适用于客服助手、企业机器人、固定人设助手等。

官方建议包括：

- 用 system prompt 设定角色
- 给出足够详细的角色背景
- 提供常见场景下的示范回复

你要注意的是，角色一致性不是一句“你是客服”就能解决的。  
真正稳定的角色输出，来自：

- 清晰边界
- 稳定示例
- 场景覆盖
- 长期上下文一致

### 9. 这篇文档和前一篇 Prompt 文档的关系

前一篇 `Prompting best practices` 更像总原则；  
这一篇 `Increase Consistency` 更像稳定性专题。

两篇合在一起，你应该形成这样的工程意识：

- Prompt 定义目标和边界
- 示例提高复现度
- 检索固定知识基础
- 任务拆链降低复杂度
- Structured Outputs 负责严格结构

这才是一个稳系统，而不是只靠一句 Prompt 撑住所有问题。

## 关键术语

- `consistency`：相同任务下输出风格、结构和行为的稳定程度
- `Structured Outputs`：通过 schema 约束模型输出格式的能力
- `few-shot examples`：用少量输入输出示例引导模型模式的方式
- `retrieval`：检索外部知识后再让模型基于检索结果回答
- `prompt chaining`：把复杂任务拆成多步 Prompt 流程
- `role consistency`：模型在多轮或多次运行中保持角色口径一致

## 练习题

1. 为什么 Anthropic 会先强调“严格 JSON 问题应该交给 Structured Outputs”？
2. 一致性至少可以拆成哪两类？它们分别应该靠什么方法解决？
3. 如果你发现模型格式总漂，为什么“继续多写几句要求”不一定有效？
4. retrieval 为什么不仅能补知识，还能提升一致性？
5. 什么样的任务应该用 prompt chaining？
6. 为什么角色一致性不能只靠一句简单的人设描述？
7. 请你设计一个学习笔记整理 Prompt，并说明你会如何加入示例来提升一致性。

## 学完后的自测标准

- 你能解释行为一致性和结构一致性的区别
- 你能说出 Anthropic 提到的 6 种提升一致性的方法
- 你能判断什么时候该继续优化 Prompt，什么时候该直接上 Structured Outputs
