# Repository Guidelines

## 项目结构与内容组织
这是一个以 AI 学习与资料沉淀为主的仓库，维护时优先关注 [`doc/`](./doc)。当前学习笔记、周计划和阶段性目标主要放在 `doc/` 目录下，其中 `doc/01-week/` 用于按周拆分学习内容。[`app/`](./app) 目前仅保留少量示例代码，不应作为仓库的主要组织中心。

新增内容时，优先按主题或时间维度放入 `doc/`，例如周计划、学习总结、术语整理、实践记录。`doc/01-week/01_day/` 这类目录用于集中存放某一天的资料清单与对应学习笔记。除非有明确需要，不要提交 `.idea/`、`.venv/`、`.uv-cache/` 等本地环境文件。

## 构建、测试与开发命令
仓库包含少量 Python/FastAPI 示例代码，依赖通过 `pyproject.toml` 管理，开发时可使用 `uv`。

```bash
uv sync
uv run uvicorn app.main:app --reload
uv run python app/main.py
```

`uv sync` 用于安装或同步示例代码依赖。`uv run uvicorn app.main:app --reload` 用于本地启动 FastAPI 示例服务并开启热重载。`uv run python app/main.py` 会通过模块中的 `__main__` 入口启动同一服务。如果只是维护学习文档，通常不需要运行这些命令。

## 编码风格与命名约定
文档是本仓库的核心内容。编写 Markdown 时，使用清晰的层级标题，段落简短，尽量按“目标、内容、结论、下一步”组织信息。文件名应可读、可检索，并与所在目录的命名风格保持一致。

现有文档同时包含数字前缀命名（如 `01_day.md`）和中文标题命名，新增文件应优先沿用所在目录的既有风格，不要随意重命名已有资料。若修改 `app/` 中的示例代码，仍遵循 Python 3.9+ 与 PEP 8：4 空格缩进，函数和变量使用 `snake_case`。

## 学习笔记规范
新增学习笔记时，优先基于官方文档、官方教程或一手资料整理，不要只做二手摘要。若用户给出明确网页链接，应先阅读该链接，再生成笔记。笔记内容应尽量详细，目标是支持系统学习，而不是只保留简短总结。

每篇学习笔记默认包含以下部分：

- `资料来源`
- `学习目标`
- `笔记概要`
- `详细笔记`
- `关键术语`
- `练习题`
- `学完后的自测标准`

`详细笔记` 应尽量贴近原文主线，说明章节结构、核心概念、概念之间的关系、为什么重要、适用场景和常见误区。写法上优先做“解释”和“串联”，不要只罗列结论。`练习题` 应覆盖理解题、对比题和应用题，便于用户学完后立即检查掌握程度。

## 文档维护建议
更新学习资料时，优先保证内容可追踪、可复习。建议在文档中写清日期、学习主题、关键结论和待办事项；引用外部资料时，附上来源名称或链接。对同一主题的持续学习，优先在原目录下增量整理，而不是创建大量分散文件。

如果某个资料清单文档已经列出多篇文章或教程，例如 `01_day.md` 中的表格，应在同目录按顺序补齐对应学习笔记，并保持统一命名和统一结构，方便后续连续学习。

如果新增实践代码或示例接口，建议补充最基本的验证方式。仓库当前尚未提交正式测试套件；后续如增加测试，推荐使用 `pytest`，并通过 `uv run pytest` 执行。

## 提交与合并请求规范
当前提交历史采用带作用域前缀的简短格式，例如 `[w1-d1] 代码矿建搭建`。后续提交建议继续使用这种格式：先写方括号作用域，再写简洁摘要，并保持一次提交只处理一个逻辑变更。

提交 Pull Request 时，应优先说明本次新增或整理了哪些学习内容、涉及哪些文档路径、为什么要调整结构。如果修改了示例代码，再补充手动验证步骤和示例请求响应。若有对应学习计划、任务记录或阶段目标，也应一并关联。
